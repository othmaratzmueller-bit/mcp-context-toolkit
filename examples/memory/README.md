# Memory store — example structure

This mirrors the on-disk shape of a `mcp-context-toolkit` memory store. Point
`CONTEXT_MEMORY_DIR` at a dir laid out like this.

## How you write a memory (atomic)

You capture knowledge as one small markdown file per fact, with frontmatter.
The **filename stem is the stable slug**; `name` may be a human title.

```markdown
---
name: prefer_composition
description: Prefer composition over deep inheritance in this codebase.
metadata:
  type: feedback          # user | feedback | project | reference | misc
tags: [design]
---

Deep inheritance chains here caused fragile-base-class bugs twice. Default to
composition / small interfaces. Link related notes with [[their-name]].
```

`recall(query)` ranks these by keyword × frecency. The frontmatter `description`
feeds ranking and the generated catalog.

## How it consolidates (packages)

When the store grows, `/context-classify` bundles atomic memories into **thematic
packages** (this dir's `core/`, `project/`, …). A package is one file holding many
members as `## slug` sections — body **verbatim**, lossless. See `core/output_style.md`.

## The 3-layer read model (all generated, never hand-edited)

| Layer | File | Loaded |
| --- | --- | --- |
| Hot index | `MEMORY.md` | always — packages by tier, tiny |
| Catalog | `_descriptions.md` | on demand — every member → its package |
| Full text | `core/*.md`, `project/*.md`, … | on demand — the package itself |

`MEMORY.md` + `_descriptions.md` are **build artifacts** (regenerated by the
indexer / `/context-dream`). Edit the source files, not these.
